# Домашнее задание №4 — Notification Platform

| | |
|---|---|
| **Курс** | SOA / Архитектура |
| **Тема** | Централизованная платформа уведомлений онлайн-банка |
| **Дата** | 14.06.2026 |

---

## Контекст и бизнес-цели

**Проблема сегодня:** команды (платежи, переводы, кредиты, маркетинг) отправляют уведомления самостоятельно → задержки, дубли, нет единого контроля, критичные сообщения теряются.

**Бизнес-цели:**

| Цель | Метрика успеха |
|------|----------------|
| Retention | **+15%** за 12 месяцев после централизации |
| Снижение жалоб | **−30%** обращений «не пришло / пришло дважды» |
| Гарантия критичных | **≥ 99.99%** транзакционных уведомлений доставлены через ≥ 1 канал |

**Каналы доставки:** push, SMS, email.  
**Типы уведомлений:** транзакционные (критичные), сервисные, маркетинговые.

**Масштаб (из задания):**

| Параметр | Значение |
|----------|----------|
| MAU | 10 000 000 |
| DAU | 3 000 000 |
| Peak CCU | 300 000 |
| Уведомлений на пользователя / день | 10 (2 + 3 + 5) |
| Маркетинговая кампания (max) | 1 000 000 получателей |

## Глоссарий

| Термин | Определение |
|--------|-------------|
| **Транзакционное уведомление** | Критичное сообщение о финансовой операции (перевод, списание, подтверждение) |
| **Failover** | Автоматическое переключение на резервный канал при ошибке/timeout основного |
| **Delivery** | Одна логическая попытка доставить одно уведомление пользователю (может включать несколько channel attempts) |
| **ASR** | Architecturally Significant Requirement — требование, определяющее архитектурные решения |

## 1. Функциональные требования

> На уровне поведения системы, без технических деталей, связаны с бизнес-целями.

| ID | User Story / Требование | Критерий приёмки | Бизнес-цель |
|----|-------------------------|------------------|-------------|
| **FR-1** | Как **пользователь**, я хочу получать подтверждение перевода и списания **сразу после операции**, чтобы быть уверенным, что операция прошла. | Подтверждение приходит в течение нескольких секунд после успешной операции | Гарантия критичных; −30% жалоб |
| **FR-2** | Как **пользователь**, я хочу **настраивать** сервисные и маркетинговые уведомления по каналам, при этом **транзакционные всегда доставляются**. | Транзакционные нельзя отключить; сервисные — частично; маркетинговые — полностью | Retention +15%; −30% жалоб |
| **FR-3** | Как **продуктовая команда**, я хочу запускать маркетинговую кампанию на **до 1 млн пользователей** через единый API. | Кампания принимается и исполняется без прямых интеграций с провайдерами | Централизация |
| **FR-4** | Как **пользователь**, я хочу получать **ровно одно** уведомление по событию, даже если платформа переключилась на другой канал. | При failover пользователь видит одно сообщение, а не push + SMS одновременно | −30% жалоб |
| **FR-5** | Как **оператор поддержки**, я хочу видеть **статус доставки** по ID операции (отправлено / доставлено / ошибка / канал). | Статус доступен по `event_id` или `delivery_id` за ≤ 2 с | −30% жалоб |
| **FR-6** | Как **команда кредитов**, я хочу отправлять сервисные уведомления о статусе заявки с учётом **предпочтительного канала** пользователя. | Статус заявки доставляется в preferred channel, если он доступен | Retention +15% |
| **FR-7** | Как **пользователь**, я хочу получать акции и кэшбэк в **едином формате** и только в разрешённых каналах. | Шаблоны согласованы; отключённые каналы не используются | Retention +15% |
| **FR-8** | Как **команда платежей**, я хочу, чтобы повторная отправка того же события **не создавала** второе уведомление пользователю. | Повтор `event_id` не приводит к повторной доставке | −30% жалоб; доверие |

### Политика отключения уведомлений (авторское решение)

| Тип | Можно отключить? | Комментарий |
|-----|------------------|-------------|
| **Транзакционные** | ❌ Нет | Перевод, списание, подтверждение операции, подозрительная активность |
| **Сервисные** | ⚠️ Частично | По каналам; минимум 1 канал для статуса заявки остаётся |
| **Маркетинговые** | ✅ Полностью | Все каналы, включая push/SMS/email |

## 2. Нефункциональные требования

> Архитектурного уровня, измеримые метрики.

| ID | Категория | Требование | Метрика / SLO |
|----|-----------|------------|---------------|
| **NFR-1** | Performance | Низкая задержка транзакционных | P99 «событие → первая попытка доставки» ≤ **3 с**; P99 end-to-end (с failover) ≤ **15 с** |
| **NFR-2** | Scalability | Массовые кампании | 1 млн получателей за ≤ **4 ч** без деградации транзакционного SLA |
| **NFR-3** | Availability | Доступность ingress | **99.95%** uptime API / месяц; RPO ≤ **1 мин**, RTO ≤ **15 мин** |
| **NFR-4** | Reliability | Гарантия критичных | ≥ **99.99%** транзакционных доставлены через ≥ 1 канал / 30 дней |
| **NFR-5** | Correctness | Дедупликация | ≤ **0.01%** дубликатов одного логического уведомления |
| **NFR-6** | Cost | Эффективность каналов | SMS ≤ **15%** от всех доставок; push — primary для некритичных |
| **NFR-7** | Observability | Наблюдаемость | 100% транзакционных в end-to-end trace; MTTD инцидента ≤ **5 мин** |
| **NFR-8** | Security | Защита данных | PII в payload шифруется at-rest; TTL хранения контента ≤ **90 дней** |

### Расчёт нагрузки платформы (порядок величин)

**Исходные данные:** DAU = 3M, 10 уведомлений / user / day.

| Тип | / user / day | / day | Avg RPS | Peak RPS (×10) |
|-----|-------------|-------|---------|----------------|
| Транзакционные | 2 | 6 000 000 | 69 | **700** |
| Сервисные | 3 | 9 000 000 | 104 | 1 040 |
| Маркетинговые | 5 | 15 000 000 | 174 | 1 740 |
| **Итого** | **10** | **30 000 000** | **347** | **~3 500** |

**Кампания 1M:** burst ≈ 1M / (4×3600) ≈ **70 RPS** sustained или **500–1000 RPS** при агрессивном burst — требует отдельной marketing-очереди.

**Вывод:** транзакционный контур (~700 RPS peak) должен быть **физически изолирован** от маркетингового (~1740 RPS peak).

|---|---|

## 4. Ключевые архитектурные вопросы

### Q-1: Как изолировать транзакционный и маркетинговый трафик?

| | |
|---|---|
| **Порождают** | ASR-1, ASR-3, NFR-1, NFR-2 |
| **Почему важен** | Peak marketing ~1740 RPS vs transactional ~700 RPS. Общая очередь → head-of-line blocking → нарушение SLA переводов. |

---

### Q-2: Кто владеет orchestration failover — центральный оркестратор или choreography?

| | |
|---|---|
| **Порождают** | ASR-2, FR-4, FR-5, NFR-5 |
| **Почему важен** | От этого зависят dedup, race conditions, возможность Support API показать единый статус. |

---

### Q-3: Когда считать push «не доставленным» и запускать failover?

| | |
|---|---|
| **Порождают** | ASR-2, NFR-4, NFR-5 |
| **Почему важен** | Слишком рано → лишние SMS (cost + жалобы). Слишком поздно → нарушение SLA. |

---

### Q-4: Как совместить user preferences и обязательность транзакционных?

| | |
|---|---|
| **Порождают** | FR-2, ASR-2, ASR-4 |
| **Почему важен** | Ошибка → регуляторный риск (не доставили) или финансовый (SMS всем). |

## 5. Архитектурные последствия ASR

### ASR-1: Низкая задержка

- Отдельный Kafka topic / queue `critical.notifications` с dedicated consumer group
- Priority scheduling: transactional > service > marketing
- Hot path: Gateway → Orchestrator → Kafka → Adapter (≤ 3 sync hop'а до enqueue)
- Redis cache для user preferences (TTL 5 min), без синхронного вызова Profile Service
- Push как primary channel (cheapest + fastest)
- Colocation orchestrator + gateway в одном K8s namespace / AZ для ≤ 1 ms network

### ASR-2: Гарантированная доставка + failover

- **Delivery Orchestrator** с explicit state machine (PENDING → SENT → DELIVERED / ESCALATED → FAILED)
- PostgreSQL — source of truth для `deliveries` + `delivery_attempts`
- Idempotency key = `event_id` (Redis + unique DB index)
- Failover policy: push (8s timeout, 2 retry) → SMS (30s, 2 retry) → email (60s, 1 retry)
- Callback Ingress для provider webhooks (DLR, push receipts)
- DLQ + PagerDuty для terminal FAILED
- Dedup guard: adapter проверяет `status != DELIVERED` перед send

### ASR-3: Массовые кампании

- Topic `marketing.campaigns` с sharding по `campaign_id`
- Bulk workers с configurable rate (respect provider limits)
- Campaign Scheduler — запуск in off-peak или throttled
- Circuit breaker на каждом provider adapter
- Resource quotas: marketing pods не используют critical node pool

### ASR-4: Cost optimization

- Policy Engine: channel ranking per notification type
- SMS только после failed push (critical) или explicit opt-in (marketing)
- Email batching для non-urgent service notifications
- Metric `sms_cost_ratio` + alert при превышении 25% для critical path

### ASR-5: Observability

- `delivery_id` = root trace ID (OpenTelemetry)
- Append-only таблица `delivery_events` для audit
- Grafana dashboards: latency P50/P99, failover rate, SMS ratio, error budget burn
- Structured logging с `event_id`, `user_id`, `channel`, `attempt`

## 6. Архитектурные решения, которые НЕ подходят

### ❌ Решение 1: Единая FIFO-очередь для всех типов

| | |
|---|---|
| **Нарушает** | ASR-1, ASR-3 |
| **Почему** | Кампания 1M сообщений блокирует head-of-line. P99 latency переводов → минуты/часы вместо секунд. |

### ❌ Решение 2: Синхронный вызов провайдера из Payment Service

| | |
|---|---|
| **Нарушает** | ASR-2, ASR-5, FR-3, FR-8 |
| **Почему** | Нет failover, dedup, audit. Payment Service блокируется на недоступность SMS-шлюза. Возврат к «разрозненным уведомлениям». |

### ❌ Решение 3: Fire-and-forget без персистентного состояния

| | |
|---|---|
| **Нарушает** | ASR-2, NFR-4 |
| **Почему** | Crash воркера = потеря сообщения навсегда. Невозможно 99.99% reliability и расследование инцидентов. |

### ❌ Решение 4: SMS как primary channel для всех критичных

| | |
|---|---|
| **Нарушает** | ASR-1, ASR-4, NFR-6 |
| **Почему** | SMS медленнее push (~2–5 с vs ~1 с), в 50–100× дороже. При 6M transactional/day → ~$180K/day vs ~$0 push. |

## 7. Неопределённости и архитектурные риски

### U-1: Реальная deliverability push

| | |
|---|---|
| **Неизвестно** | % устройств без token, Do Not Disturb, задержки FCM/APNs в peak |
| **Как проверить** | Pilot 1–5% DAU; метрики `accepted → displayed`; A/B timeout 5/8/15 с |
| **Risk** | Aggressive failover → SMS cost spike |

### U-2: Регуляторные требования к SMS для крупных операций

| | |
|---|---|
| **Неизвестно** | Обязателен ли SMS для переводов > N руб. независимо от push |
| **Как проверить** | Legal review; feature flag `regulatory_sms_threshold` в Policy Engine |
| **Risk** | Compliance violation или лишние SMS |

### U-3: Конкуренция failover и late push delivery

| | |
|---|---|
| **Неизвестно** | Как часто push приходит после timeout (ghost delivery) |
| **Как проверить** | Log analysis `push.delivered_after_escalation`; suppress duplicate UI-side |
| **Risk** | Push + SMS одновременно → жалобы (FR-4) |

## 8. RFC — гарантированная доставка с кросс-канальным failover

> **Полный документ:** [`RFC-critical-delivery-failover.md`](./RFC-critical-delivery-failover.md)  
> **Диаграммы PlantUML:** [`diagrams/`](./diagrams/)  
> **Диаграммы PNG (готовы к просмотру):** [`diagrams/png/`](./diagrams/png/)

### Резюме

Подсистема **Critical Delivery Service (CDS)** — state machine + orchestrator + channel adapters.

| | Вариант A: Orchestrator | Вариант B: Choreography |
|---|------------------------|------------------------|
| **Суть** | Центральный orchestrator владеет state | Coordinator реагирует на Kafka events |
| **Dedup** | ✅ Single owner | ⚠️ Redis locks, race risk |
| **Latency** | ✅ Меньше hop'ов | ⚠️ +100–300 ms |
| **Support API** | ✅ PG = SoT | ⚠️ Projection lag |
| **Рекомендация** | ✅ **Выбран** | Альтернатива |

**Failover policy:** push (8s) → SMS (30s) → email (60s).  
**Decision:** Вариант A — проще ASR-2 (dedup), FR-5 (audit), FR-4 (no duplicate UX).
